\begin{takeaway}
\textbf{Headline:} In the ten days since the course submission, AttnFuse
has grown from a forward-only inference DSL into a complete attention
training-and-inference system. It now matches PyTorch SDPA within
\textbf{1.06$\times$} on a real Llama-3-8B training step, wins all 12
KV-cache decoding head-to-heads against PyTorch \texttt{flex\_attention},
beats \texttt{flex\_attention} on every RoPE-composition cell by up to
\textbf{2.59$\times$}, and supports BigBird-style block-sparse
\emph{training} -- a capability not currently available in any
mainstream Triton-based attention compiler.
\end{takeaway}

\section{What was added}
%% =============================================================

The course submission shipped: forward-only attention, five variants,
three dtypes, three baselines, tile ablation, roofline, fused RoPE,
25 GPU tests. The work since then breaks into four buckets.

\paragraph{Training.} Implemented the full FlashAttention-2-style
backward pass. The forward optionally writes the per-row log-sum-exp
to a small fp32 buffer; the backward runs three kernels (preproc for
$D = \mathrm{rowsum}(\mathrm{d}O \odot O)$, then dK/dV, then dQ), all
hitting tensor cores via \texttt{tl.dot}. Wired into PyTorch via a
\texttt{torch.autograd.Function} so any AttnFuse call with
\texttt{requires\_grad} automatically uses the new path. Verified
correct on 30 new tests covering dense, causal, ALiBi, and GQA / MQA
geometries across fp16 / bf16 / fp32.

\paragraph{LLM inference (KV-cache decoding).} The standard attention
kernel under-utilises the GPU for autoregressive decoding (one new
token $Q.N=1$) because it launches only $B \cdot H_q$ programs. Added
a two-phase Flash Decoding kernel that splits the KV axis across many
parallel programs and combines their partial $(m, \ell, \mathrm{acc})$
tuples via the standard log-sum-exp algebra. A second iteration uses
tensor-core matmuls and \emph{cyclic Q-head replication} to share K/V
loads inside a GQA group without losing tensor-core throughput. Across
12 (Llama-3-8B / Llama-3-70B / Falcon-MQA) $\times$ (1k / 4k / 16k / 32k)
geometries, AttnFuse \textbf{beats \texttt{flex\_attention} in every
cell} (Fig.~\ref{fig:kvcache}).

\paragraph{Block-sparse attention (BigBird-style).} Added a new
\texttt{MaskKind.BLOCK\_SPARSE} backed by a dedicated kernel that
iterates over a CSR-style active-block list. The user constructs a
\texttt{BlockMask} from a Python predicate via
\texttt{af.create\_block\_mask(predicate, Q\_LEN, KV\_LEN, BLOCK\_M, BLOCK\_N)}.
At 7.7\% active blocks (BigBird pattern: global rows + global cols +
local window), the kernel runs $7.4\times$ faster than dense AttnFuse
at $N=4096$ -- genuine sub-quadratic FLOPs. A subsequent commit added
the matching \emph{backward} kernels, so block-sparse \emph{training}
also works.

\paragraph{HuggingFace integration.} Registered AttnFuse as a
HuggingFace attention backend (\texttt{attn\_implementation="attnfuse"}).
The integration supports GQA natively (no \texttt{repeat\_kv} expansion
since AttnFuse broadcasts via its \texttt{GROUP\_SIZE} constexpr) and
auto-routes single-token decoding to the Flash Decoding fast path. A
real Llama-3-8B-class \texttt{LlamaDecoderLayer} runs forward and full
forward+backward+\texttt{optimizer.step()} at \textbf{1.06$\times$ of
PyTorch SDPA} on the 3090; \texttt{flex\_attention} OOMs at
HEAD\_DIM=128 on the 3090 (its tile config requires 131~KB SMEM but
the 3090 has only 101~KB available).

\paragraph{Other improvements.} GQA / MQA support across every variant;
cross-attention with $N_q \neq N_{kv}$ for encoder--decoder and
KV-cache decoding; H100 (sm\_90) detection and tile tables with
deploy scripts ready for SSH; a Hypothesis-driven correctness fuzzer
(80 randomised configurations per run, caught one real bug); honest
benchmarks against PyTorch \texttt{flex\_attention} in every category.

\subsection{Forward: RoPE compositions (the structural-novelty case)}

\texttt{flex\_attention}'s \texttt{score\_mod} runs on the already-computed
$QK^\top$ tile, so RoPE -- which must rotate $Q$ and $K$ \emph{before}
the matmul -- cannot be fused. AttnFuse fuses everything in one
kernel; \texttt{flex\_attention} pays for two extra host-side rotation
kernels plus an HBM round-trip.

\section{Engineering notes}
%% =============================================================

\subsection{Triton 3.1 fp32 \texttt{tl.dot} workaround}

The backward dQ matmul hit a Triton 3.1 bug:
\texttt{tl.dot}$(\mathrm{ds}, K)$ in fp32 silently produces wrong
results when the first operand is computed in registers (rather than
freshly loaded from HBM). All three precision modes (default tf32,
\texttt{input\_precision='ieee'}, \texttt{tf32x3}) hit the same bug
inside our kernel; in a standalone matmul outside this context, all
three work correctly. The workaround -- casting $\mathrm{ds}$ to
bf16 before \texttt{tl.dot} -- recovers tensor-core throughput with
mantissa-level precision loss (well under FA2 tolerance). The bf16
cast is annotated in the kernel source with the full diagnostic path
so the next Triton release can be re-tested.

\subsection{Hopper (H100, sm\_90) port}

The compiler auto-detects sm\_90 at import time and switches to
Hopper-specific tile tables (\texttt{\_HOPPER\_TABLE\_F16},
\texttt{\_HOPPER\_TABLE\_F16\_SPARSE}, etc.) that exploit the larger
228~KB SMEM (vs Ampere's 101~KB) and deeper pipeline stages. Deploy
scripts at \texttt{scripts/setup\_h100.sh} and
\texttt{scripts/run\_h100\_benchmarks.sh} make a one-command SSH
deployment possible; the benchmark sweep will refine the tile tables
once hardware time is available.

\section{Open items}
%% =============================================================

\begin{description}[leftmargin=1.5em,itemsep=2pt]
  \item[H100 sweep] Run the deploy scripts on H100 hardware, capture
    paper-grade numbers, re-tune the Hopper tile tables based on
    measured perf. \emph{Expected:} closing the residual 4--6\% gap to
    \texttt{flex\_attention} on dense and causal at N=4096, because
    WGMMA tensor cores have a smaller Triton-vs-CUTLASS overhead than
    Ampere's standard MMA path.
  \item[Triton 3.2+] Re-test the bf16 cast once Triton fixes the fp32
    \texttt{tl.dot} bug. Lossless backward in fp32 becomes available
    at no semantic cost.
  \item[Ring attention] Multi-GPU sequence parallelism. The IR is
    structurally ready; lowering to a sequence-parallel kernel is the
    biggest remaining swing for the paper.
  \item[fp8 / int8 quantisation] Production-deployment story; one new
    constexpr-gated dtype path in the existing kernel template.
\end{description}
